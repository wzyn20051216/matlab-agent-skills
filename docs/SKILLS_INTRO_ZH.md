# MATLAB Agent Skills 项目介绍

`matlab-agent-skills` 是一个面向工程研发场景的 MATLAB / Simulink 智能技能仓库，目标不是“让 AI 多写几段 `.m` 代码”，而是把 MATLAB 变成一个可调度、可验证、可复现、可交付的工程执行环境。

它服务的不是演示型问答，而是真正的工程闭环：接收任务、选择合适能力、调用 MATLAB 执行、生成日志与产物、完成验收，并把结果沉淀为后续可复用的自动化流程。

---

## 我们为什么要做这个项目

MATLAB 本身已经非常强大，覆盖数据分析、建模仿真、控制优化、信号与图像处理、代码生成、硬件部署、机器人、自主系统、深度学习等大量工程场景。但在真实研发流程里，工程师仍然经常面临几个痛点：

- MATLAB 能力很多，但入口分散，复杂任务难以快速选路。
- AI 能写代码，但经常停留在“看起来对”，无法形成可执行闭环。
- 仿真、代码生成、工具链联调、论文复现往往需要多人手工串联。
- 结果缺少统一日志、产物目录、版本信息与验收记录，难复查、难交接、难自动化。

`matlab-agent-skills` 的意义，就是把这些离散能力封装成一套真正可落地的工程技能系统，让 Codex 或其他智能代理不只是“会说 MATLAB”，而是“真的能把 MATLAB 工程跑起来”。

---

## 这个仓库到底是什么

可以把它理解为一套 MATLAB 工程智能中台，核心由三层组成：

### 1. 技能层

把常见任务拆成明确技能，而不是按工具箱名字机械分类。当前已经覆盖：

- `matlab-orchestrator`：任务入口与路由中枢。
- `matlab-runner`：MATLAB 批处理执行、日志采集、产物落盘。
- `matlab-data-analysis`：数据处理、统计分析、拟合建模、图表导出。
- `matlab-simulink-modeling`：Simulink 模型创建、修改、仿真、结果检查。
- `matlab-codegen-deploy`：MEX、MATLAB Coder、Simulink Coder、部署链路。
- `matlab-control-optimization`：控制、辨识、优化与自动调参。
- `matlab-robotics-autonomy`：机器人、ROS、导航、自主系统工作流。
- `matlab-signal-vision-ai`：信号处理、图像视觉、深度学习与 AI 应用。
- `matlab-testing-ci`：单元测试、验收测试、CI 验证与报告输出。

### 2. 执行层

通过 PowerShell 脚本和 MATLAB 批处理机制，把技能变成真正可运行命令，而不是停留在说明文档。每次执行都尽量保证：

- 有明确入口命令；
- 有可追溯日志；
- 有可检查产物；
- 有 JSON 或结构化验收结果；
- 有失败时可定位的问题线索。

### 3. 验证层

这个仓库最强调的一点，是“任务必须闭环验收”。所以我们不是只给技能定义，还配了本地验证脚本、功能探针、编译链 smoke test、硬件支持检查和嵌入式工具链检查。一个技能值不值钱，不在于名字多酷，而在于它能不能稳定跑通。

## 它解决了什么问题

对于嵌入式、电力电子、电气、自动化、计算机等工程师，这个仓库的价值主要体现在五件事：

### 第一，降低复杂 MATLAB 工程的操作门槛

很多工程任务并不是不会做，而是流程太长：先查工具箱、再找脚本、再补依赖、再跑仿真、再导出结果、再看日志、再手工验收。现在这些步骤可以被统一封装成技能和脚本，显著减少重复劳动。

### 第二，让 AI 输出从“答案”升级为“结果”

传统 AI 更像咨询顾问，能给建议；`matlab-agent-skills` 更像执行工程师，强调把建议落实到 MATLAB 实际运行结果、图表、代码生成产物、验证报告与自动验收结论。

### 第三，支持工程任务的可复现与可交接

不论是复现一篇开源论文、跑一个控制系统仿真、导出一版代码生成结果，还是做一条嵌入式联调链，我们都希望过程和结果能留下痕迹。这样团队内二次复查、版本切换、对比分析才有基础。

### 第四，为新版本 MATLAB 能力建立“可信入口”

MATLAB 新版本能力越来越多，但是否真的能在本机环境、真实工具链、真实工程里稳定工作，不能靠想象。这个仓库的思路是：新功能可以积极拥抱，但必须通过探针、测试和验收证明“能用”。

### 第五，打通从算法到部署的链路

仓库已经把 MATLAB、Simulink、代码生成、STM32、Raspberry Pi、CubeMX、Keil、VS Code 等环节纳入统一验收视角。它不只是写算法，也关注算法如何落到工具链、落到部署、落到工程系统。

## 我们当前已经做到了什么

当前仓库已经完成了一套本地可验证的基础闭环，重点包括：

- MATLAB 主程序可从终端稳定拉起；
- 工具箱清单可导出；
- 特性探针可输出结构化报告；
- 数值拟合类任务可自动检查结果误差；
- 图片与数据产物可自动导出并检查非空；
- MATLAB Coder 可生成 C 代码产物；
- Simulink 最小模型可创建、保存与仿真；
- Simulink Coder 最小代码生成链可验证；
- STM32 与 Raspberry Pi 的 MATLAB 硬件支持可探测；
- VS Code、STM32CubeMX、Keil 等嵌入式工具链可发现并验收。

这意味着，仓库已经不只是“想法集合”，而是具备实际执行力的工程自动化骨架。

## 这和普通的 MATLAB 提示词、脚本仓库有什么不同

市面上很多 MATLAB AI 项目，更多是“提示词 + 示例代码 + 工具箱说明”的组合，适合学习，但离工程落地还有距离。`matlab-agent-skills` 的差异主要有三点：

- **不是只回答问题，而是推动执行。**
- **不是只生成代码，而是追求验收。**
- **不是只展示能力，而是强调复现。**

我们更在乎一件事：一个工程师把任务交给代理之后，是否真的能得到可运行、可检查、可保存、可复盘的结果。

## 典型应用场景

### 1. 算法与建模工程

适合做数据处理、参数辨识、统计分析、控制器设计、约束优化、批量试验和图表自动导出。

### 2. Simulink 与模型驱动开发

适合做模型搭建、参数扫描、结果对比、最小模型验证、代码生成预检查，以及模型级回归验收。

### 3. 嵌入式部署与联调准备

适合做 MEX / C 代码生成、外部工具链检查、STM32 支持环境探测、Raspberry Pi 支持探测，以及部署前的一键验收。

### 4. 开源论文与项目复现

适合记录论文链接、代码仓库、数据集、MATLAB 版本、工具箱依赖、复现步骤、数值偏差和最终产物，形成规范的复现实验流。

### 5. 团队级工程自动化

适合把个人经验沉淀为团队级技能，统一日志结构、产物目录、脚本入口和 CI 验收方式，减少“只在某个人电脑上能跑”的风险。

## 面向工程师，我们坚持什么原则

### 真执行，而不是假懂

不会把“看起来合理”的回答当成完成任务。能运行就运行，能验收就验收，能输出报告就输出报告。

### 真复现，而不是讲故事

对于论文复现、开源项目复现、代码生成、仿真结果检查，优先保留命令、版本、日志、清单和差异结果。

### 真闭环，而不是只到一半

任务不是生成了脚本就结束，而是要尽可能走到执行结果、图表、代码、报告、工具链检查和可交付结论。

### 真面向工程，而不是只面向演示

我们重点服务的是工程研发链路，因此会优先考虑工具链兼容性、可维护性、可扩展性和后续自动化空间。
